# Room Booking Service

Сервис для бронирования переговорных комнат с JWT-аутентификацией, ролями `admin` и `user`, расписаниями, генерацией слотов и бронированием конкретных временных окон.

## Что умеет API

Основные группы ручек:
- `/_info` — технический health endpoint
- `/dummyLogin` — быстрый тестовый вход под `admin` или `user`
- `/register`, `/login` — обычная регистрация и логин пользователя
- `/rooms/list`, `/rooms/create` — просмотр и создание переговорных комнат
- `/rooms/{roomId}/schedule/create` — создание расписания для комнаты
- `/rooms/{roomId}/slots/list` — получение доступных слотов на дату
- `/bookings/create`, `/bookings/my`, `/bookings/list`, `/bookings/{bookingId}/cancel` — создание, просмотр и отмена бронирований
- `/swagger/index.html` — Swagger UI для HTTP API

## Основные фичи

- разграничение доступа по ролям `admin` и `user`
- JWT-аутентификация для защищенных эндпоинтов
- одна переговорка может иметь только одно расписание
- слоты фиксированы по 30 минут
- слоты генерируются заранее на 7 дней и лениво догенерируются при запросе более дальних дат
- горизонт запросов по слотам ограничен 30 днями вперед
- защищенность от двойного бронирования за счет бизнес-проверок и ограничений PostgreSQL
- опциональное создание `conferenceLink` при бронировании через мок внешнего сервиса
- Swagger-документация, Makefile-команды, сидирование тестовых данных и нагрузочное тестирование hot endpoint

## Быстрый запуск

Нужен файл `.env` в корне проекта. Минимальный пример:

```env
APP_HTTP_PORT=8080
APP_JWT_SECRET=replace-with-long-random-secret
APP_JWT_TTL_MINUTES=60

POSTGRES_DB=room_booking
POSTGRES_USER=admin
POSTGRES_PASSWORD=replace-with-strong-password
POSTGRES_PORT=5432

APP_POSTGRES_DSN=postgres://admin:replace-with-strong-password@localhost:5432/room_booking?sslmode=disable
APP_POSTGRES_TEST_DSN=postgres://postgres:postgres@localhost:55432/room_booking_test?sslmode=disable
```

Где `APP_POSTGRES_DSN` должен ссылаться на ту же основную БД, что описана через `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` и `POSTGRES_PORT`. То есть данные внутри URL в `APP_POSTGRES_DSN` должны соответствовать этим же env-переменным.

`APP_POSTGRES_TEST_DSN` менее критичен: он нужен только для integration tests и может указывать на отдельную test DB с любыми подходящими тестовыми параметрами.

ps: в автооценке системы не прокидываются переменные окружения, поэтому пришлось писать дефолтные значения в docker-compose

Запуск сервиса через Makefile:

```bash
make up
```
Тестовые данные можно создать через makefile:

```bash
make seed
```

`make seed` нужно вызывать после `make up`, так как он использует уже поднятую основную БД и текущие значения из `.env`.

После старта сервис доступен на `http://localhost:8080`.

Какие контейнеры поднимутся:
- `postgres`
- `migrate` — накатывает `migrations/000001_init.up.sql`
- `app`

Чтобы положить сервер с сохранением данных в бд на следующие запуски:
```bash
make down
```
Чтобы положить сервер с удалением данных в бд на следующие запуски:
```bash
make down-v
```
## Swagger

make swagger

## Тесты
```bash
make test
```

Алгоритм `make test`:
- поднимается одноразовый контейнер `postgres:17`
- в нем создается test DB на `localhost:55432`
- запускаются unit и integration tests
- считается coverage
- контейнер удаляется

На данный момент не кросспакетный coverage проекта  44.5%, а кросспакетный 67.8%.

`make test` использует отдельный одноразовый контейнер `postgres:17` для интеграционных тестов. Он сам поднимает test DB на `localhost:55432`, прогоняет тесты и затем удаляет контейнер.

Локальный Postgres для `make test` не нужен.

После выполнения `make test` создается файл `coverage.out`.

## Нагрузочное тестирование
Для нагрузочного тестирования нужен установленный `k6`:

```bash
sudo snap install k6
```

Запуск нагрузочного тестирования hot endpoint:

```bash
make perf-test-slots
```

Алгоритм `make perf-test-slots`:
- поднимается отдельный контейнер `postgres:17` для perf DB
- накатываются миграции
- запускается seed
- поднимается отдельный экземпляр приложения на `localhost:8081`
- запускается `k6`-сценарий для `slots/list`
- после завершения perf-контур останавливается

Краткий результат локального прогона hot endpoint `/rooms/{roomId}/slots/list`:
- инструмент: `k6`
- профиль нагрузки: `50 VUs`, `30s`
- `http_req_failed = 0.00%`
- `avg = 12.47ms`
- `p95 = 16.12ms`

## Основные решения

### Архитектура

Проект разделен на слои:
- `internal/domain` — сущности и инварианты
- `internal/application` — use cases и порты
- `internal/infrastructure` — HTTP, Postgres, JWT, bcrypt, clock, id generator
- `cmd/server` — composition root

Так соблюдается чистая архитектура, а бизнес-правила не смешиваются с транспортом и БД, а use case'ы тестируются отдельно.

### Генерация слотов

Для генерации слотов выбрана комбинированная стратегия, кратко:
- при создании расписания слоты генерируются на ближайшие 7 дней вперед
- если пользователь запрашивает дату, на которую слоты еще не были созданы, они лениво генерируются при первом запросе
- горизонт запросов ограничен 30 днями вперед

Это помогает не плодить слишком много слотов, но и расширить доступность.
### Ограничения консистентности

Ключевые ограничения дублируются на уровне БД:
- одно расписание на комнату
- уникальный слот по `room_id + start_at`
- только одна активная бронь на слот через partial unique index

Это защищает от гонок не только в коде, но и в PostgreSQL.

## Решение по `conferenceLink`

Текущее поведение:
- если `createConferenceLink=false`, бронь создается как обычно
- если `createConferenceLink=true`, сервис сначала запрашивает ссылку у внешнего `Conference Service`, затем создает бронь и сохраняет ссылку в записи брони
- в проекте используется мок, который возвращает детерминированную ссылку вида `https://conference.local/rooms/<bookingId>`

### Что решили делать при сбоях

#### 1. Внешний сервис недоступен или возвращает ошибку до ответа

Решение: бронь не создается, клиент получает ошибку.

Почему так:
- пользователь явно запросил бронь со ссылкой на конференцию
- бронировать без ссылки было бы неочевидно для пользователя
- так сохраняется простая и честная семантика: либо бронь создана вместе со ссылкой, либо операция целиком не выполнена

#### 2. Ошибка после успешного ответа внешнего сервиса, но до сохранения брони в БД

Текущее решение: запрос завершается ошибкой, запись брони в БД не появляется

Почему так:
- для тестового задания делать явный retry механизм было бы избыточно да и ни к чему, поэтому выбран простой и явный компромисс


### OpenAPI и кодогенерация

HTTP API описывается через `api.yaml`, а transport-слой частично генерируется через `oapi-codegen`.

Что генерируется:
- DTO и схемы запросов/ответов
- `ServerInterface`
- server wrapper

Что пишется руками:
- адаптер между generated API и use case'ами
- middleware
- mapping ошибок

Почему так:
- снижается риск расхождения со спецификацией
- контракт API остается первичным
- в HTTP-слое меньше ручной рутины

### Router на chi

В качестве HTTP router выбран `chi`, а не `gin` и не чистый `net/http`.

Почему так:
- `chi` легкий и близок к стандартной библиотеке
- хорошо сочетается с generated server wrapper
- middleware и роутинг получаются достаточно удобными, но без лишней магии

### UTC и модель времени

Все timestamp'ы, относящиеся к слотам и API-операциям, трактуются строго в UTC.

Дополнительно для расписания используется отдельный тип `TimeOfDay`, а не `time.Time`.

Почему так:
- это соответствует условию задачи
- уменьшается риск ошибок из-за timezone
- расписание задается как время внутри дня (`HH:MM`), а не как абсолютный timestamp

### Слот как единица бронирования

`Slot` трактуется как фиксированный интервал ровно 30 минут, а не как произвольное окно бронирования.

Следствие:
- одна бронь = один слот
- длинные бронирования как набор нескольких слотов сейчас не поддерживаются отдельным сценарием

Почему так:
- условие явно задает фиксированную длину слота 30 минут
- API бронирует именно `slotId`
- это хорошо ложится на индексы, защиту от гонок и список доступности
